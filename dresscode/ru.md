# Dresscode — alfaCTF 2026

**Категория:** Crypto / Web  
**Сложность:** Hard  
**Теги:** crypto, web, sqli, aes-cbc, mac-malleability, iv-flip, race  
**Флаг:** `alfa{5mO7rYA_KAK01_fAbR1C_SMotryA_sK0lKO_D3TAIl5}`

## Описание

Онлайн-магазин одежды в стиле Gorpcore. При регистрации начисляется 100 кредитов. Для получения флага нужно собрать вещи из всех 8 категорий (jacket, pants, shorts, tshirt, sneakers, cap, gloves, accessories) и пройти проверку дресс-кода. Самый дешёвый комплект стоит 703 кредита — в 7 раз больше стартового баланса. Пополнения нет.

## Архитектура

| Сервис | Роль |
|--------|------|
| **shop** (Flask) | Веб-фронтенд. Шифрует транзакции и записывает в MySQL. |
| **validator** (Rust) | Опрашивает MySQL каждые 3 секунды, проверяет MAC, расшифровывает AES-CBC, выполняет бизнес-логику (регистрация / покупка). |
| **MySQL** | Общее хранилище. У `shop_user` есть SELECT/INSERT/UPDATE на таблицу `transactions`. |

Покупки работают асинхронно: магазин шифрует данные заказа AES-256-CBC, вычисляет HMAC-SHA256, вставляет строку в `transactions`. Валидатор забирает, проверяет, обрабатывает.

## Уязвимость

### 1. MAC не покрывает IV (крипто)

```python
# shop/app.py:121-129
cipher = AES.new(AES_KEY, AES.MODE_CBC, iv)
ciphertext = cipher.encrypt(pad(plaintext, AES.block_size))
mac = hmac.new(MAC_KEY, ciphertext, hashlib.sha256).hexdigest()
```

HMAC считается **только от шифротекста**, IV хранится отдельно и **не** аутентифицируется. В режиме CBC изменение бита IV приводит к изменению того же бита в **первых 16 байтах** открытого текста, не затрагивая последующие блоки и MAC.

### 2. SQL-инъекция в `update_comment` (веб)

```python
# shop/app.py:533
query = f"UPDATE transactions SET comment = '{new_comment}' WHERE id = '{order_id}'"
```

Параметр `comment` подставляется в UPDATE без экранирования. Это позволяет модифицировать **любой столбец** строки транзакции, включая `iv`.

### 3. Обход для аккаунта владельца (логика)

```rust
// validator/src/main.rs:283
if data.from == String::from("1000001") {
    // пропуск списания баланса
}
```

Аккаунт владельца (`AlexCab`, id `1000001`) имеет баланс 999 999 999 и освобождён от списания. Если `data.from` после расшифровки равен `1000001`, любая сумма проходит проверку — деньги не списываются.

## Эксплуатация — CBC IV Flip через SQLi

Открытый текст транзакции покупки (`json.dumps`):

```
{"from": "DDDDDDD", "to": "DDDDDDD", "items": [...], "amount": ...}
```

Первые 16 байт: `{"from": "DDDDDD` (байты 0–15), где `D` — первые 6 цифр 7-значного user_id. Байт 16 (7-я цифра) **не изменяется** при модификации IV.

**Цель:** подменить байты 10–15 открытого текста на `100000`, чтобы поле `from` читалось как `"1000001"`. Для этого последняя цифра нашего user_id должна быть `1`.

### Шаги

1. **Регистрируем аккаунты**, пока не получим `user_id`, оканчивающийся на `1` (~10 % вероятность на каждую попытку).
2. **Добавляем в корзину** по одной самой дешёвой вещи из каждой из 8 категорий (8 предметов, 703 кредита).
3. **POST `/checkout`** — создаёт необработанную транзакцию. Из редиректа извлекаем `order_id`.
4. **Немедленно POST `/orders/<order_id>/update_comment`** с SQL-инъекцией, которая XOR'ит IV побайтово прямо в MySQL:

   ```sql
   x', iv = CONCAT(
     SUBSTR(iv,1,20),
     LPAD(HEX(CONV(SUBSTR(iv,21,2),16,10) ^ d10),2,'0'),
     LPAD(HEX(CONV(SUBSTR(iv,23,2),16,10) ^ d11),2,'0'),
     LPAD(HEX(CONV(SUBSTR(iv,25,2),16,10) ^ d12),2,'0'),
     LPAD(HEX(CONV(SUBSTR(iv,27,2),16,10) ^ d13),2,'0'),
     LPAD(HEX(CONV(SUBSTR(iv,29,2),16,10) ^ d14),2,'0'),
     LPAD(HEX(CONV(SUBSTR(iv,31,2),16,10) ^ d15),2,'0')
   ) WHERE id = '<order_id>' -- 
   ```

   где `d10…d15 = ord(uid[i]) ^ ord(target[i])` для шести изменяемых позиций. XOR вычисляется внутри MySQL — оригинальный IV читать не нужно.

5. **Валидатор** подхватывает модифицированную транзакцию через ~3 с:
   - MAC проходит (шифротекст не менялся).
   - Расшифровка с новым IV даёт `"from": "1000001"`.
   - Проверка баланса проходит (у владельца 999M).
   - Списание пропускается (особый случай для владельца).
   - Вещи добавляются в **наш** инвентарь (`"to"` за пределами байта 15, не изменён).
6. **GET `/check_dresscode`** — все 8 категорий есть, флаг выдан.

### Гонка (race condition)

Валидатор опрашивает БД каждые 3 секунды. SQL-инъекцию нужно выполнить **до** того, как он заберёт транзакцию. Поскольку `/checkout` возвращает ответ мгновенно (не ждёт обработки), а сетевой round-trip ~100–250 мс, окно более чем достаточное.

## Главные выводы

- «Encrypt-then-MAC» защищает только то, что покрыто MAC. Если IV хранится отдельно от шифротекста, первые 16 байт открытого текста при CBC остаются податливыми независимо от стойкости MAC: `HMAC(ciphertext)` ≠ `HMAC(IV ‖ ciphertext)`.
- SQLi, не достающая до таблиц `users` или `flags`, — не «низкий impact». Когда таблица очереди хранит криптоматериал (IV, nonce, соль), даже UPDATE-примитив превращается в инструмент подделки для каждой обрабатываемой записи.
- Привилегированные fast-path'ы для «владельца» внутри асинхронных воркеров опасны, когда воркер аутентифицирует запрос только по расшифрованному полю. Проверка должна быть на *подписанном* конверте, а не на *расшифрованном* содержимом.
