## Описание

Дан SSH-доступ к серверу с bash-приложением для создания тир-листов (`redflag.sh`). Приложение использует `whiptail` для TUI-меню. Пользователи могут создавать тир-листы, добавлять тиры/элементы, присоединяться к чужим тир-листам и просматривать публичные. Некоторые элементы помечены как `[LOCKED]` через поле `restricted=` в файле `.meta` каждого тир-листа — просматривать их может только владелец.

Цель: прочитать заблокированный элемент внутри тир-листа, принадлежащего `nastya_gorp` — зарезервированному пользователю, под которым нельзя войти.

## Уязвимость

Функция `join_tierlist_by_name` переименовывает директорию тир-листа, добавляя суффикс `__with__<username>`:

```bash
mv "$TIERLISTS_DIR/$tl_name" "$TIERLISTS_DIR/${tl_name}__with__${CURRENT_USER}"
```

**Не проверяется**, существует ли директория назначения. В Linux `mv src dst`, где `dst` — существующая директория, перемещает `src` **внутрь** `dst` (т.е. `dst/src`), а не перезаписывает.

Два дополнительных фактора завершают эксплойт:

1. `view_tierlist` использует **рекурсивный** `find` без `-maxdepth`:
   ```bash
   find "$tl_dir" -type d -name 'tier_*' -print0
   ```
   Это находит директории тиров на любой глубине вложенности — включая чужой тир-лист, перемещённый внутрь нашего.
2. `is_restricted` читает `.meta` **внешней** директории (`$tl_dir`), а не вложенной оригинальной. Поскольку в нашем `.meta` поле `restricted=` пустое, заблокированные элементы из вложенного тир-листа отображаются без ограничений.

## Эксплуатация

1. Подключаемся: `ssh redflag@redflag-h306xs1x.alfactf.ru`.
2. Входим под именем `pwn` (любое валидное незарезервированное имя).
3. Заходим в **«Публичные тир-листы»** и узнаём название цели: `layering_tierlist` (владелец `@nastya_gorp`).
4. **Создаём** свой тир-лист с именем `layering_tierlist__with__pwn` — это заранее создаёт директорию назначения.
5. Возвращаемся в публичные тир-листы, выбираем `layering_tierlist`, жмём **«Присоединиться»**.
6. Скрипт выполняет `mv .../layering_tierlist .../layering_tierlist__with__pwn`. Так как директория назначения существует, `layering_tierlist` перемещается **внутрь** неё.
7. Теперь в просмотре `layering_tierlist__with__pwn` мы — владелец. Рекурсивный `find` находит `layering_tierlist__with__pwn/layering_tierlist/tier_*/` и отображает все элементы — включая ранее заблокированный — используя **наш** `.meta` (без ограничений).
8. Кликаем на элемент и читаем флаг.

## Главные выводы

- `mv src existing_dir/` — это «переместить внутрь», а не «переименовать»: в сочетании с предсказуемыми именами атакующий заранее создаёт директорию назначения и встраивает свои метаданные поверх чужих.
- Рекурсивный обход (`find` без `-maxdepth`) доверяет внешнему `.meta`. Когда проверка доступа читает `/path/.meta`, а перечисление — `/path/**/items/*`, контролируемое атакующим внешнее состояние молча перекрывает внутреннее состояние жертвы.
- «Зарезервированные» системные пользователи не дают безопасности, если атакующий может слить состояние жертвы в свой namespace.
